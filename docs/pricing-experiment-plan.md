# 価格実験計画

## 概要

リサーチで推奨されたA/Bテストを実装計画に落とし込み、収益最大化のための価格・機能・導線の最適化を段階的に実施する。

---

## 実験の優先順位

### Phase 1: MVP期（1-2ヶ月目）

**目標**：トラクション獲得（収益化は後回し）

- 全機能無料提供
- KPI計測基盤の構築
- バイラルループの検証

**実施しない実験**：価格実験（データが不十分なため）

---

### Phase 2: ソフトローンチ期（3ヶ月目）

**目標**：初期収益化テスト

**実験1-4を順次実施**

---

### Phase 3: 本格収益化期（4-6ヶ月目）

**目標**：持続可能な収益モデルの確立

**実験5-8を順次実施**

## 実験一覧

### 実験1：ペイウォールの位置

**仮説**：
- ペイウォールを結果ページ直後に置くと、初回意思決定（Day0）で課金率が上がる
- サブスクアプリのベンチマークでは「トライアルの80-90%がDay0に発生」

**A案：無料要約→ペイウォール**

- 診断完了後、3行要約のみ表示
- 詳細レポートはペイウォール内

```
┌─────────────────┐
│ 診断完了       │
└─────────────────┘
         │
         ▼
┌─────────────────┐
│ 3行要約        │
│ ・内省的       │
│ ・戦略的       │
│ ・完璧主義     │
└─────────────────┘
         │
         ▼
┌─────────────────┐
│ ペイウォール    │
│ 「詳細を見る」 │
│ ¥980/月〜     │
└─────────────────┘
```

**B案：途中チラ見せ→結果直後ペイウォール**

- 診断完了後、詳細レポートの「目次」と「1章だけ」を表示
- 続きはペイウォール内

```
┌─────────────────┐
│ 診断完了       │
└─────────────────┘
         │
         ▼
┌─────────────────┐
│ 目次           │
│ 第1章：特性    │（← 表示）
│ 第2章：失敗パターン │（← ロック）
│ 第3章：相性    │（← ロック）
│ 第4章：テンプレ │（← ロック）
│ 第5章：行動計画 │（← ロック）
└─────────────────┘
         │
         ▼
┌─────────────────┐
│ ペイウォール    │
│ 「続きを読む」 │
│ ¥980/月〜     │
└─────────────────┘
```

**主KPI**：
- 結果UU → 課金率（CVR）

**副KPI**：
- 完了率（診断開始→完了）
- 離脱率（ペイウォール表示→離脱）

**期間**：2週間

**サンプルサイズ**：
- 各群1,000 UU以上（統計的有意性確保のため）

**判定基準**：
- CVRが0.5pt以上高い方を採用（例：A案1.0% vs B案1.5%）

**実装**：
- Next.js App Router の Dynamic Rendering
- A/Bテストツール（PostHog / Vercel Edge Config）

### 実験2：トライアル日数

**仮説**：
- 長いトライアルが高転換という示唆がある一方、最適日数はプロダクト依存
- 性格診断+AI相談は「すぐ価値がわかる」ため、短期トライアルで十分かもしれない

**A案：3日間無料トライアル**

**B案：7日間無料トライアル**

**主KPI**：
- トライアル開始率（結果UU → トライアル開始）
- トライアル → 課金率

**副KPI**：
- トライアル中の利用回数（AI相談回数）
- 解約率（トライアル後30日）

**期間**：2週間

**サンプルサイズ**：
- 各群500トライアル開始以上

**判定基準**：
- 純増MRR（トライアル開始率 × 課金率 × 単価 - 解約損失）で判定

**実装**：
```typescript
// Stripe Checkout Session
const session = await stripe.checkout.sessions.create({
  mode: 'subscription',
  subscription_data: {
    trial_period_days: variant === 'A' ? 3 : 7,
  },
  // ...
})
```

### 実験3：価格階層

**仮説**：
- 単一価格（980円）は「高い」層を取りこぼす
- 2段階（ライト240円 / 標準980円）なら、ライトで初回ハードルを下げつつ、ARPPUも確保できる

**A案：単一価格（月額980円）**

```
┌─────────────────┐
│ FREE            │
│ 0円             │
│ ・診断          │
│ ・AI相談月3回  │
└─────────────────┘
         │
         ▼
┌─────────────────┐
│ PLUS            │
│ 980円/月        │
│ ・AI相談月20回 │
│ ・週次サマリー │
└─────────────────┘
```

**B案：2段階価格（ライト240円 / 標準980円）**

```
┌─────────────────┐
│ FREE            │
│ 0円             │
│ ・診断          │
│ ・AI相談月3回  │
└─────────────────┘
         │
         ├───────────┐
         ▼           ▼
┌─────────────┐ ┌─────────────┐
│ LITE        │ │ PLUS        │
│ 240円/月    │ │ 980円/月    │
│ ・AI相談月10回│ │ ・AI相談月20回│
│             │ │ ・週次サマリー│
└─────────────┘ └─────────────┘
```

**主KPI**：
- 課金率（結果UU → 課金）
- ARPPU（課金ユーザー単価）

**副KPI**：
- プラン別構成比（A案：100% PLUS、B案：LITE vs PLUS）
- 返品/返金率

**期間**：4週間（価格変更は学習期間が必要）

**サンプルサイズ**：
- 各群2,000 UU以上

**判定基準**：
- 総収益（課金率 × ARPPU）で判定
- 例：A案（CVR 1.5% × 980円 = 14.7円/UU）vs B案（CVR 2.5% × 600円 = 15.0円/UU）

**実装**：
- Stripe Price ID の切り替え
- UI上のプラン選択画面

### 実験4：共有の有料/無料

**仮説**：
- 共有カードを有料にすると、課金率は上がるが新規流入が減る
- 共有は無料、詳細レポートは有料、という線引きがバランスが良い

**A案：共有カード有料**

- 診断完了後、結果は無料で閲覧可能
- 共有カード（1:1 / 9:16）生成は有料（単発980円 or サブスク）

**B案：共有は無料、詳細は有料**

- 診断完了後、結果+共有カードは無料
- 詳細レポート（PDF / 相性深掘り）は有料

**主KPI**：
- 新規流入UU（共有経由の診断開始）

**副KPI**：
- 課金率
- share/user（共有率）

**期間**：2週間

**サンプルサイズ**：
- 各群1,000 UU以上

**判定基準**：
- LTV（新規流入UU × 課金率 × ARPPU）で判定
- バイラルループが壊れると長期的損失が大きいため、新規流入UUを重視

### 実験5：AI相談の回数制限 vs 使い放題

**仮説**：
- 使い放題は魅力的だが、重課金ユーザーで粗利が毀損する
- 回数制限+クレジット追加の混合モデルが粗利を守る

**A案：PLUS 使い放題（980円/月）**

**B案：PLUS 月20回まで（980円/月）+ クレジット追加（980円/20回）**

**主KPI**：
- 粗利率（収益 - AI APIコスト）

**副KPI**：
- クレジット購入率（B案のみ）
- 解約率

**期間**：4週間

**サンプルサイズ**：
- 各群500課金ユーザー以上

**判定基準**：
- 粗利/ユーザーで判定
- 例：A案（980円 - 400円APIコスト = 580円）vs B案（980円 - 200円APIコスト + 100円クレジット収益 = 880円）

### 実験6：年払い割引率

**仮説**：
- 年払いはLTV向上とチャーン低下に寄与
- 割引率が高すぎると収益減、低すぎると転換しない

**A案：年払い20%割引（7,800円/年、月換算650円）**

**B案：年払い30%割引（6,500円/年、月換算542円）**

**主KPI**：
- 年払い比率（月払い vs 年払い）

**副KPI**：
- 年次継続率（1年後の継続）
- LTV

**期間**：8週間（年払いは判断が遅い）

**サンプルサイズ**：
- 各群500課金ユーザー以上

**判定基準**：
- LTV（年払い比率 × 年額 × 継続年数）で判定

---

### 実験7：無料枠の広さ

**仮説**：
- 無料を薄くすると課金率は上がるが、獲得効率が下がる
- 無料を厚くするとバイラルは強まるが、課金理由が薄れる

**A案：無料は診断のみ（AI相談なし）**

**B案：無料は診断+AI相談月3回**（現行案）

**主KPI**：
- 課金率

**副KPI**：
- 診断完了率
- share/user
- 紹介経由比率

**期間**：4週間

**サンプルサイズ**：
- 各群2,000 UU以上

**判定基準**：
- 総収益/UU で判定（獲得効率と課金率の両方を考慮）

---

### 実験8：単発課金の価格

**仮説**：
- 詳細レポート1,980円は、就活層には「安い」が、娯楽層には「高い」
- 価格を下げると購入率は上がるが、単価が下がる

**A案：詳細レポート 1,980円**

**B案：詳細レポート 1,280円**

**主KPI**：
- 単発購入率（結果UU → 購入）
- ARPPU（単発購入者の平均単価）

**副KPI**：
- 購入者の属性（年齢・用途）

**期間**：3週間

**サンプルサイズ**：
- 各群1,000 UU以上

**判定基準**：
- 総収益（購入率 × 単価）で判定

### 2. 実装

**ツール**：
- **PostHog**：Feature Flags + A/Bテスト + ファネル分析
- **Vercel Edge Config**（代替案）：エッジでA/B割り当て

**実装例**（PostHog）：
```typescript
// app/tests/[testId]/result/[resultId]/page.tsx
'use client'

import { useFeatureFlagVariantKey } from 'posthog-js/react'

export default function ResultPage() {
  const variant = useFeatureFlagVariantKey('paywall-position')

  if (variant === 'control') {
    // A案：無料要約→ペイウォール
    return <ResultSummaryOnly />
  } else {
    // B案：途中チラ見せ→結果直後ペイウォール
    return <ResultWithPreview />
  }
}
```

### 4. 分析

**統計的有意性の確認**：
- **有意水準**：α = 0.05（5%）
- **検出力**：β = 0.8（80%）
- **最小効果量**：CVR 0.5pt差（例：1.0% vs 1.5%）

**必要サンプルサイズ計算**：
```
n = 2 × (Zα/2 + Zβ)² × p(1-p) / (p1 - p2)²

例：p = 0.015（ベースCVR 1.5%）、p1-p2 = 0.005（0.5pt差）
→ n ≈ 1,500 per group
```

**PostHog Experiment Results**：
- 自動で有意性検定
- Confidence Interval表示

### 5. 判定

**判定会議**（週次）：
- KPI達成状況
- 統計的有意性
- 定性フィードバック（ユーザーサポート等）

**意思決定**：
- 勝者を本番採用
- または追加実験（A/B/Cテスト）

## リスク管理

### 実験失敗のリスク

**リスク1：課金率が大幅に下がる**

**対策**：
- 段階的ロールアウト（10% → 50% → 100%）
- 即座にロールバック可能な設計

**リスク2：バイラルループが壊れる**

**対策**：
- 共有機能は常に無料を維持（実験4で確認）
- share/userを常時監視

**リスク3：チャーンが急増**

**対策**：
- トライアル設計を慎重に（実験2）
- 解約理由のフィードバック収集

### 法務リスク

**リスク：価格表示・返金ポリシーの不備**

**対策**：
- 特定商取引法に基づく表示
- 返金ポリシーの明記
- 自動更新の事前通知
